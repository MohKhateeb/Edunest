# 🔄 خريطة سير العملية - Booking Flow

## المسارات الرئيسية

### **السيناريو الأول: حجز عادي مدفوع - النسخة السعيدة ✅**

```
┌─────────────────────────────────────────────────────────────────────┐
│                    HAPPY PATH - سيناريو مثالي                       │
└─────────────────────────────────────────────────────────────────────┘

  ولي الأمر
    │
    └─→ 📝 ينقر "حجز جديد"
         ├─ يختار الطالب ✓
         ├─ يختار المعلم والخدمة ✓
         ├─ يختار الوقت (بعد ساعتين) ✓
         ├─ يضيف ملاحظات إضافية (اختياري)
         └─ ينقر "تأكيد الحجز"
              │
              └─→ Backend:
                   ├─ ✓ التحقق من توفر المعلم
                   ├─ ✓ التحقق من توثيق المعلم
                   ├─ ✓ حساب السعر والعمولة
                   ├─ CREATE Booking (PENDING, UNPAID)
                   ├─ CREATE Payment (isPaid: false)
                   ├─ NOTIFY المعلم: "حجز جديد ينتظر قبول"
                   └─ RETURN bookingId
                        │
                        └─→ UI: "تم الحجز! الرجاء دفع المبلغ"
                             │
  ولي الأمر                  │
    │                        │
    └─→ 💳 ينقر "دفع الآن"  │
         ├─ يفتح PaymentModal
         ├─ يدخل بيانات البطاقة
         └─ ينقر "تأكيد الدفع"
              │
              └─→ Backend (processPayment):
                   ├─ ✓ التحقق من ملكية الحجز
                   ├─ ✓ التحقق من حالة الحجز (PENDING)
                   ├─ UPDATE Payment (isPaid: true, paidAt: now)
                   ├─ UPDATE Booking (paymentStatus: PAID)
                   ├─ NOTIFY المعلم: "تم الدفع! يمكنك قبول الحجز"
                   └─ RETURN success
                        │
                        └─→ UI: "تم الدفع بنجاح"
                             │
  المعلم                     │
    │                        │
    └─→ 📱 يفتح تطبيقه     │
         ├─ يرى إشعار: "حجز جديد"
         ├─ ينقر على BookingCard
         ├─ يراجع تفاصيل الحجز
         └─ ينقر "✓ قبول"
              │
              └─→ Backend (acceptBooking):
                   ├─ ✓ التحقق من ملكية الخدمة
                   ├─ ✓ التحقق من الدفع (PAID)
                   ├─ UPDATE Booking (status: CONFIRMED, meetingUrl: ...)
                   ├─ NOTIFY ولي الأمر: "المعلم قبل الحجز"
                   └─ RETURN success
                        │
                        └─→ UI: "تم قبول الحجز"
                             │
                             │ (ينتظر موعد الجلسة)
                             │
  المعلم + ولي الأمر        │
    │                        │
    ├─→ 📞 في موعد الجلسة  │
    │   ├─ ينقران على الرابط (Jitsi meet)
    │   ├─ ينضمان إلى الجلسة
    │   └─ تقديم الشرح
    │
    └─→ 🎓 انتهاء الجلسة
         │
  المعلم                    │
    │                       │
    └─→ 📄 بعد 30 دقيقة  │
         ├─ ينقر "رفع تقرير"
         ├─ يملأ النموذج:
         │  ├─ هل حضر الطالب؟ نعم
         │  ├─ المواضيع: "المعادلات الخطية"
         │  ├─ الأداء: 4/5
         │  ├─ الواجب: "10 مسائل صفحة 45"
         │  └─ ملاحظات: "متفاعل وممتاز"
         └─ ينقر "حفظ التقرير"
              │
              └─→ Backend (submitSessionReport):
                   ├─ ✓ فحص حالة الجلسة (grace_period/expired)
                   ├─ CREATE SessionReport
                   ├─ UPDATE Booking (status: COMPLETED)
                   ├─ INCREMENT Teacher.totalSessions
                   ├─ NOTIFY ولي الأمر: "التقرير جاهز"
                   └─ RETURN success
                        │
                        └─→ UI: "تم حفظ التقرير"
                             │
  (بعد 24 ساعة - في دورة الراتب)
    │
  الإدارة                   │
    │                       │
    └─→ 💰 في لوحة التحكم │
         ├─ AdminPayoutsEngine
         ├─ تحديد فترة التسوية
         ├─ ينقر "حساب التسوية"
         └─ ينقر "إصدار تسوية"
              │
              └─→ Backend (createTeacherPayout):
                   ├─ ✓ جلب الحجوزات المستحقة
                   ├─ ✓ حساب الأرباح الصافية
                   ├─ ✓ التحقق من عدم التداخل
                   ├─ CREATE TeacherPayout
                   ├─ UPDATE Booking.payoutId (ربط)
                   └─ RETURN success
                        │
                        └─→ UI: "تم إصدار التسوية"
                             │
  المعلم                    │
    │                       │
    └─→ 💵 يرى أرباحه    │
         ├─ لوحة تحكم المعلم
         └─ الدخل الشهري: 700 ريال
              │
  الإدارة                   │
    │                       │
    └─→ 🏦 تحويل بنكي    │
         ├─ ينقر "تأكيد الدفع"
         ├─ يختار طريقة التحويل
         └─ ينقر "تحويل"
              │
              └─→ Backend (markPayoutAsPaid):
                   ├─ UPDATE TeacherPayout (isPaid: true, paidAt: now)
                   ├─ NOTIFY المعلم: "تم تحويل الأرباح!"
                   └─ RETURN success
                        │
                        └─→ UI: "تم التحويل بنجاح"
                             │
  المعلم                    │
    │                       │
    └─→ 💰 يستقبل التحويل │
         └─ 700 ريال في حسابه البنكي ✅

═══════════════════════════════════════════════════════════════════════
النتيجة النهائية:
  • الطالب: تعليم جيد + تقرير مفصل
  • المعلم: 700 ريال (بدون عمولة المنصة)
  • المنصة: عمولة من السعر الأصلي
  • ولي الأمر: اطمئنان على تقدم ابنه
═══════════════════════════════════════════════════════════════════════
```

## **السيناريو الثاني: إلغاء مبكر - استرجاع أموال ✅**

```
┌─────────────────────────────────────────────────────────────────────┐
│          CANCELLATION PATH - الإلغاء المبكر يستحق استرجاع          │
└─────────────────────────────────────────────────────────────────────┘

  Booking: PENDING → PAID (ولي الأمر دفع)
  ⏱ 30 ساعة قبل الجلسة المقررة
    │
  ولي الأمر
    │
    └─→ ❌ ينقر "إلغاء الحجز"
         ├─ يختار السبب: "تغير البرنامج"
         ├─ ينقر "تأكيد الإلغاء"
         └─ يرى رسالة: "سيتم استرجاع المبلغ خلال 3-5 أيام"
              │
              └─→ Backend (cancelBooking):
                   ├─ ✓ التحقق: أم = ولي أمر
                   ├─ ✓ التحقق: حالة الحجز = PENDING ✓
                   ├─ ✓ التحقق: متبقي قبل الجلسة = 30 ساعة (> 24) ✓
                   ├─ ✓ عدد الاسترجاعات: 1/2 ✓
                   │
                   ├─ UPDATE Booking:
                   │  ├─ status: CANCELLED
                   │  ├─ paymentStatus: REFUNDED
                   │  ├─ cancelledBy: userId
                   │  ├─ cancelledByType: PARENT
                   │  └─ cancellationReason: "تغير البرنامج"
                   │
                   ├─ UPDATE Payment:
                   │  └─ isPaid: false
                   │
                   ├─ CREATE ParentRefund:
                   │  ├─ amount: 100
                   │  ├─ isPaid: false (ينتظر الإدارة)
                   │  └─ createdAt: now
                   │
                   ├─ NOTIFY ولي الأمر:
                   │  └─ "تم إلغاء الحجز - الاسترجاع قيد المعالجة"
                   │
                   ├─ NOTIFY المعلم:
                   │  └─ "تم إلغاء الحجز - الطلب من ولي الأمر"
                   │
                   └─ NOTIFY الإدارة:
                      └─ "طلب استرجاع أموال: 100 ريال"
                           │
  (في لوحة تحكم الإدارة)     │
    │                        │

    │                       │
    └─→ 💰 AdminPaymentsQueue
         ├─ ترى طلب استرجاع جديد
         ├─ تراجع البيانات
         ├─ تتحقق من المستند
         ├─ ينقر "✓ تأكيد التحويل"
         └─ تختار طريقة التحويل
              │
              └─→ Backend (markParentRefundAsPaid):
                   ├─ UPDATE ParentRefund:

                   │  └─ paidAt: now
                   │
                   └─ NOTIFY ولي الأمر:
                      └─ "✅ تم تحويل المبلغ!"
                           │
  ولي الأمر                 │
    │                       │
    └─→ 💵 يستقبل التحويل │
         └─ 100 ريال في حسابه البنكي ✅

═══════════════════════════════════════════════════════════════════════
الخلاصة:
  ✅ لم يحدث شيء غير المتوقع
  ✅ الاسترجاع تلقائي
```

## **السيناريو الثالث: نزاع على الجلسة - الحل 🔥**

```
┌─────────────────────────────────────────────────────────────────────┐
│            DISPUTE PATH - الاعتراض على الجلسة المكتملة              │
└─────────────────────────────────────────────────────────────────────┘

  Booking: COMPLETED + Payment: PAID
  ⏱ بعد انتهاء الجلسة بـ 2 ساعة
    │
  ولي الأمر
    │
    └─→ ⚠️ يرى التقرير:
         ├─ "المواضيع: لا شيء"
         ├─ "الأداء: 1/5"
         └─ يشعر: "هذا خطأ! لم يقدم المعلم شيء!"
              │
              └─→ ينقر "⚠️ فتح اعتراض"
                   ├─ يختار السبب: "المعلم لم يقدم الخدمة كما وعد"
                   ├─ يفسر: "لم يشرح شيئاً - أضاع الوقت فقط"
                   └─ ينقر "تقديم"
                        │
                        └─→ Backend (createDispute):
                             ├─ ✓ فحص: Booking.status = COMPLETED ✓
                             ├─ ✓ فحص: لا يوجد نزاع سابق ✓
                             ├─ ✓ فحص: completedAt < 24 ساعة ✓
                             ├─ ✓ فحص: payoutId = null (لم تتم التسوية) ✓
                             │
                             ├─ CREATE Dispute:
                             │  ├─ status: OPEN
                             │  ├─ parentUserId: userId
                             │  ├─ reason: "المعلم لم يقدم الخدمة..."
                             │  └─ allowedTurn: BOTH
                             │
                             ├─ CREATE DisputeMessage (رسالة فتح):
                             │  └─ "[رسالة نظام] المعلم لم يقدم الخدمة..."
                             │
                             ├─ NOTIFY المعلم:
                             │  └─ "⚠️ اعتراض جديد على الجلسة"
                             │
                             └─ NOTIFY الإدارة:
                                └─ "⚠️ نزاع جديد - يحتاج تدخل"
                                     │
  المعلم                           │
    │                              │
    └─→ 📱 يرى الإشعار          │
         ├─ ينقر على DisputeModal
         ├─ يرى شكوى ولي الأمر
         ├─ ينقر "رد"
         ├─ يكتب: "أنا شرحت بشكل جيد، الطالب لم ينتبه!"
         └─ ينقر "إرسال"
              │
              └─→ Backend (sendDisputeMessage):
                   ├─ CREATE DisputeMessage:
                   │  ├─ senderId: teacherId
                   │  └─ message: "أنا شرحت بشكل جيد..."
                   │
                   └─ NOTIFY ولي الأمر:
                      └─ "رد من المعلم على الاعتراض"
                           │
  ولي الأمر                 │
    │                       │
    └─→ 💬 يرد:            │
         ├─ "لا، أنت لم تشرح!"
         └─ ينقر "إرسال"
              │
              └─→ Backend: CREATE DisputeMessage
                   │
                   └─ NOTIFY الإدارة:
                      └─ "الاعتراض لا يزال مفتوحاً - تدخل مطلوب"
                           │
  الإدارة                   │
    │                       │
    └─→ 🏛️ في لوحة التحكم │
         ├─ ترى DisputeModal
         ├─ تراجع جميع الرسائل
         ├─ تسمع الطالب عبر مقابلة
         └─ تقرر: "ولي الأمر محق - الجودة ضعيفة"
              │
              └─→ ينقر "حل الاعتراض"
                   ├─ يختار: "RESOLVED_IN_FAVOR_OF_PARENT"
                   ├─ يكتب ملاحظات: "تم التحقق - جودة الشرح ضعيفة"
                   └─ ينقر "تأكيد"
                        │
                        └─→ Backend (resolveDispute):
                             ├─ UPDATE Dispute:
                             │  ├─ status: RESOLVED_IN_FAVOR_OF_PARENT
                             │  ├─ adminNotes: "تم التحقق..."
                             │  └─ resolvedAt: now
                             │
                             ├─ UPDATE Booking:
                             │  └─ paymentStatus: REFUNDED
                             │
                             ├─ CREATE ParentRefund:
                             │  ├─ amount: 100
                             │  ├─ isPaid: false
                             │  └─ createdAt: now
                             │
                             ├─ NOTIFY ولي الأمر:
                             │  └─ "✅ تم حل النزاع لصالحك - استرجاع كامل"
                             │
                             └─ NOTIFY المعلم:
                                └─ "تم حل النزاع ضدك - سيتم خصم المبلغ"
                                     │
  ولي الأمر                 │
    │                       │
    └─→ 💰 يستقبل التحويل │
         └─ 100 ريال ✅
              │
  المعلم                    │
    │                       │
    └─→ 0 ريال من هذه الجلسة ✗
         └─ (لأن الجودة كانت ضعيفة)

═══════════════════════════════════════════════════════════════════════
النتيجة:
  • ولي الأمر: استرجع المبلغ (العدالة! 🎯)
  • المعلم: لم يحصل على شيء (تحذير من الكيفية)
  • المنصة: لا خسارة (لم تدفع له من الأساس)
  • النظام: حل عادل ✅
═══════════════════════════════════════════════════════════════════════
```

## **السيناريو الرابع: جلسة تجريبية - خاصة 🎓**

```
┌─────────────────────────────────────────────────────────────────────┐
│         FREE TRIAL PATH - الجلسة التجريبية الأولى                   │
└─────────────────────────────────────────────────────────────────────┘

  ولي الأمر (جديد)
    │
    └─→ 📱 يدخل التطبيق
         └─ يرى: "جربة مجانية أولى - 30 دقيقة"
              │
              └─→ ينقر "حجز جلسة تجريبية"
                   ├─ يختار الطالب
                   ├─ يختار معلم
                   ├─ ينقر "تأكيد"
                   └─ بدون دفع! (سعر = 0)
                        │
                        └─→ Backend (createBooking with isTrial=true):
                             ├─ ✓ فحص: لم يستخدم جلسة تجريبية قبل ✓
                             ├─ fحساب: price = 0, paymentStatus = UNPAID
                             ├─ حساب: trialCostToPlatform = 10 (من ميزانية المنصة)
                             │
                             ├─ CREATE Booking:
                             │  ├─ isTrial: true
                             │  ├─ price: 0
                             │  ├─ paymentStatus: UNPAID (لا يوجد دفع!)
                             │  └─ trialCostToPlatform: 10
                             │
                             ├─ CREATE Payment:
                             │  └─ isPaid: true (افتراضي - لا يوجد دفع)
                             │
                             └─ NOTIFY المعلم:
                                └─ "🎓 حجز جلسة تجريبية"
                                     │
  المعلم                           │
    │                              │
    └─→ 📱 يرى الحجز              │
         └─ ينقر "✓ قبول"
              │
              └─→ Backend (acceptBooking):
                   ├─ ✓ فحص: paymentStatus = UNPAID
                   ├─ ✓ لكن isTrial = true → قبول بدون دفع ✓
                   │
                   └─ UPDATE Booking (status: CONFIRMED)
                        │
                        │ (تقديم الخدمة وكالمعتاد)
                        │
  بعد انتهاء الجلسة              │
    │                            │
  المعلم                         │
    │                            │
    └─→ 📄 رفع التقرير          │
         └─ يعتبر الطالب جديد
              │
              └─→ Backend (submitSessionReport):
                   ├─ CREATE SessionReport
                   ├─ UPDATE Booking (status: COMPLETED)
                   └─ INCREMENT Teacher.totalSessions
                        │
  (في دورة الراتب - بعد 24 ساعة) │
    │                             │
  Backend - Payout Processing     │
    │                             │
    └─→ حساب أرباح المعلم:
         ├─ for each booking:
         │   if (isTrial) {
         │     trialCompensation += 10  ← ربح من المنصة!
         │   }
         │
         ├─ netAmount = 0 - 0 + 10 = 10 ريال
         └─ المعلم يحصل على 10 ريال من المنصة؟
              │
              ⚠️ سؤال مهم: هل هذا مقصود؟

═══════════════════════════════════════════════════════════════════════
الغموض في الجلسات التجريبية:
  ❓ من يدفع الـ 10 ريال؟ (كلفة المنصة)
  ❓ هل المعلم يحصل على شيء؟
  ❓ الهدف: التعريف؟ أم دعم المعلمين الجدد؟
═══════════════════════════════════════════════════════════════════════
```

## **السيناريو الخامس: خطأ - المعلم لم يرسل تقرير ❌**

```
┌─────────────────────────────────────────────────────────────────────┐
│      MISSING REPORT PATH - المعلم نسي إرسال التقرير                │
└─────────────────────────────────────────────────────────────────────┘

  Booking: CONFIRMED → الجلسة انتهت
  ⏱ 24 ساعة بدون تقرير
    │
    │ (نظام تلقائي - cron job يومي)
    │
  Backend - handleMissingReports()
    │
    └─→ البحث عن:
         ├─ status = COMPLETED
         ├─ report = null (لا يوجد تقرير)
         ├─ completedAt <= 24 ساعة
         └─ المعلم لم يرسل تقرير
              │
              └─→ (الخطأ الحالي!)
                   ├─ canSubmitReport() يعود TRUE ✓ (يمكن الإرسال)
                   ├─ لكن بعد 24 ساعة:
                   │   sessionState = 'ghost'
                   │   canSubmitReport() يعود TRUE ✗ (خطأ!)
                   │
                   └─ النتيجة:
                      └─ المعلم يستطيع إرسال تقرير بعد أسبوع!
                           │
  (الحل المقترح)              │
    │                         │
  Backend - تحسين:            │
    │                         │
    └─→ 1️⃣ HARD DEADLINE      │
         ├─ بعد 24 ساعة من الانتهاء
         ├─ لا يمكن إرسال تقرير
         └─ canSubmitReport() يعود FALSE
              │
         2️⃣ NOTIFY المعلم      │
         ├─ بعد 23 ساعة
         ├─ "⚠️ يتبقى ساعة واحدة لإرسال التقرير"
         └─ تنبيه إشعار
              │
         3️⃣ LATE PENALTY      │
         ├─ بعد 24 ساعة
         ├─ إنشاء تقرير افتراضي
         ├─ SessionReport (studentAttended: false)
         ├─ المعلم لا يحصل على دخل من هذه الجلسة
         └─ ولي الأمر يرى: "[تقرير نظام - فشل المعلم في الإرسال]"
              │
         4️⃣ NOTIFY الجميع    │
         ├─ المعلم: "❌ لم ترسل تقرير - لا دخل من هذه الجلسة"
         ├─ ولي الأمر: "⚠️ لم يرسل المعلم تقرير الجلسة"
         └─ الإدارة: "⚠️ معلم متكرر في تأخير التقارير"
              │
              └─ تأثيرات:
                 ├─ في التسوية: الحجز لا يُدخل
                 ├─ سمعة المعلم: ضعيفة
                 └─ إجراء: تحذير + فترة تجريبية

═══════════════════════════════════════════════════════════════════════
الخلاصة المقترحة:
  ✅ فحص صارم على حالة "ghost"
  ✅ تنبيهات استباقية
  ✅ عقوبة تلقائية للتأخير
  ✅ حماية ولي الأمر والمنصة
═══════════════════════════════════════════════════════════════════════
```

## **نقاط التوقف الحرجة (Critical Points)**

```
🔴 نقاط يجب مراقبتها باستمرار:

1. PAYMENT STATUS MISMATCH
   ├─ Booking.paymentStatus ≠ Payment.isPaid
   └─ احتمال 🚨 فقدان أموال

2. ORPHAN REPORTS
   ├─ SessionReport بدون Booking.report
   └─ احتمال 🚨 خلل في الربط

3. UNLINKED PAYOUTS
   ├─ Booking بدون payoutId بعد COMPLETED
   └─ احتمال 🚨 عدم الدفع للمعلم

4. OVERLAPPING PERIODS
   ├─ فترات تسوية متداخلة
   └─ احتمال 🚨 دفع مزدوج

5. PENDING REFUNDS
   ├─ ParentRefund بـ isPaid=false أكثر من 7 أيام
   └─ احتمال 🚨 شكاوى من الآباء

6. OPEN DISPUTES
   ├─ Dispute بـ status=OPEN أكثر من 3 أيام
   └─ احتمال 🚨 تصعيد المشكلة
```
